# 领域建模

在设计过程中主动构建并打磨项目的领域模型。这是一门**主动**的功课——挑战用词、编造边界场景、在术语和决策成形的那一刻就把它们写下来。（仅仅*阅读* `CONTEXT.md` 查词不算本技能——那是任何 skill 都该有的一行习惯。本技能用于你在**改变**模型时，而不是消费模型时。）

## 文件结构

大多数仓库只有一个上下文（context）：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果仓库根目录存在 `CONTEXT-MAP.md`，说明有多个上下文。这张地图指向每个上下文的位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 上下文级决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

**懒惰地创建文件**——只在有东西要写时才创建。如果还没有 `CONTEXT.md`，等第一个术语敲定时再建；如果还没有 `docs/adr/`，等需要第一条 ADR 时再建。

## 会话期间

### 对照词汇表挑战用词

当用户使用的术语与 `CONTEXT.md` 中已有的语言冲突时，立刻指出来："词汇表里'取消'的定义是 X，但你说的好像是 Y——到底是哪个？"

### 打磨模糊措辞

当用户使用含糊或多义的词时，提出一个精确的标准术语："你说'账户'——指的是 Customer 还是 User？这俩不是一回事。"

### 用具体场景讨论

讨论领域关系时，用具体场景做压力测试。主动编造探查边界的场景，逼用户把概念之间的界线说清楚。

### 与代码交叉验证

当用户陈述某物如何工作时，检查代码是否认同。发现矛盾就摆出来："代码里取消的是整个 Order，但你刚才说可以部分取消——哪个是对的？"

### 就地更新 CONTEXT.md

术语一旦敲定，当场更新 `CONTEXT.md`。不要攒批——趁热记录。格式见 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)。

`CONTEXT.md` 里**绝不出现实现细节**。它不是 spec、不是草稿本、也不是实现决策的堆放处。它只是一本词汇表，仅此而已。

### 克制地提议 ADR

只有同时满足以下三条，才提议创建 ADR：

1. **难以逆转**——以后改主意的成本是实打实的
2. **没有上下文会显得突兀**——未来的读者会纳闷"当时为什么这么做？"
3. **真实权衡的产物**——确实存在多个候选方案，并且出于具体理由选了其中一个

三条缺任何一条，就跳过 ADR。格式见 [ADR-FORMAT.md](./ADR-FORMAT.md)。
