# ADR 格式

ADR 存放在 `docs/adr/` 下，使用顺序编号：`0001-slug.md`、`0002-slug.md`，以此类推。

`docs/adr/` 目录懒惰创建——只在需要第一条 ADR 时才建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么，我们决定了什么，为什么。}
```

就这么简单。一条 ADR 可以只有一段话。价值在于记录*做了一个决策*以及*为什么*——而不是把各个栏目填满。

## 可选栏目

只在确实增值时才加。大多数 ADR 用不到。

- **Status** frontmatter(`proposed | accepted | deprecated | superseded by ADR-NNNN`)——决策可能被重提时有用
- **Considered Options（候选方案）**——只在被否掉的方案值得记住时
- **Consequences（后果）**——只在有非显而易见的下游影响需要点出时

## 编号

扫描 `docs/adr/` 找到现有最大编号，加一。

## 什么时候提议 ADR

以下三条必须**同时**成立：

1. **难以逆转**——以后改主意的成本是实打实的
2. **没有上下文会显得突兀**——未来的读者会看着代码纳闷"当时到底为什么这么做？"
3. **真实权衡的产物**——确实存在多个候选方案，并且出于具体理由选了其中一个

如果决策容易逆转，跳过——反正你会逆转它。如果不突兀，没人会纳闷。如果没有真正的备选，除了"我们做了显而易见的事"之外没什么可记的。

### 什么配得上 ADR

- **架构形态。** "我们用 monorepo。""写模型是事件溯源的，读模型投影到 Postgres。"
- **上下文之间的集成模式。** "Ordering 和 Billing 通过领域事件通信，不走同步 HTTP。"
- **带有锁定效应的技术选型。** 数据库、消息总线、认证提供商、部署目标。不是每个库都算——只有换一个要花一个季度的才算。
- **边界与范围决策。** "客户数据归 Customer 上下文所有；其他上下文只按 ID 引用。" 明确说"不"和说"是"同样有价值。
- **刻意偏离常规路径的决定。** "我们用手写 SQL 而不是 ORM，因为 X。" 任何让合理读者会默认相反做法的决定。这类记录能拦住下一位工程师去"修复"一个刻意的选择。
- **代码里看不见的约束。** "因为合规要求我们不能用 AWS。""响应时间必须在 200ms 以内，因为合作方的 API 合同这么规定的。"
- **否决理由不显而易见的被否方案。** 如果你考虑过 GraphQL 但出于微妙理由选了 REST，记下来——不然六个月后还会有人再提 GraphQL。
