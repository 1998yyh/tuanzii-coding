# Init CLAUDE.md — 深度项目上下文初始化

## 概述

替代 Claude Code 内置的 `/init` 命令。内置 init 只做浅层的文件结构扫描和通用描述，本 skill 生成一份**工程级别**的 CLAUDE.md，覆盖：架构分层、环境特殊规范、编码约束（三层边界模型）、测试策略、文档同步机制、病例库系统。

**宣告启动:** "使用 deepinit skill 深度初始化项目上下文。"

## 核心原则

生成的 CLAUDE.md 必须遵循以下原则，缺一不可：

1. **项目特定** — 只写从代码中观察到的事实，不写通用最佳实践
2. **不重复** — 不写 README 已有的、git 历史可查的、代码注释已说明的
3. **可操作** — 每条规范要么有具体命令、要么有代码示例、要么有文件路径
4. **分层边界** — 明确"必须做 / 需确认 / 禁止"三层
5. **环境感知** — 针对部署环境特殊约束（NFS/Docker/时区/多机）单独成章

## 执行流程

### 阶段 A：环境侦察（必做）

在开始分析前获取基本信息：

```
1. 获取当前时间戳（用于文档版本标记）
2. 确认项目根目录位置
3. 检查是否已有 CLAUDE.md（增量更新 vs 全新生成）
4. 检查是否存在 .cursorrules / .cursor/rules/ / .github/copilot-instructions.md / AGENTS.md
```

### 阶段 B：项目全景扫描

按以下顺序收集信息，每步完成后记录发现：

#### B1. 技术栈识别
- 读取 `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` / `pom.xml` 等
- 确定语言版本、框架版本、核心依赖
- 识别构建工具、包管理器

#### B2. 项目结构分析
- 识别架构模式（单体/微服务/Monorepo/分层架构）
- 找出核心目录划分逻辑（按功能/按层级/按领域）
- 识别模块边界和依赖关系

#### B3. 命令与工作流
- 找出 Makefile / scripts/ / package.json scripts / docker-compose
- 提取构建、测试、部署、迁移等关键命令
- 识别 CI/CD 配置

#### B4. 环境特殊约束
- 部署环境（Docker/K8s/NAS/云服务）
- 文件系统特殊性（NFS/共享存储）
- 时区/编码/多机一致性要求
- 配置管理（环境变量/密钥管理）

#### B5. 代码约定抽取
- 从代码样本中归纳命名风格、导入顺序、日志格式
- 异常处理模式
- 注释风格和 docstring 规范
- 已有的 lint/format 配置

#### B6. 测试策略
- 测试框架和运行方式
- 测试分层（单元/集成/e2e）
- 覆盖率要求
- 测试数据管理

#### B7. 文档体系
- 现有文档结构
- 文档与代码的同步机制
- API 文档生成方式

### 阶段 C：生成 CLAUDE.md

按以下模板结构输出，**根据实际项目内容裁剪章节**（不是每个项目都需要所有章节）：

---

## CLAUDE.md 模板结构

```markdown
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 身份定义

- **角色**: [从技术栈推断的专家角色]
- **技术栈**: [语言版本 + 框架版本 + 核心依赖]
- **项目描述**: [一句话核心业务描述]

## 项目结构

[仅列出需要理解整体架构的关键目录划分，不罗列每个文件]

## 可执行命令

[实际可用的构建/测试/部署命令，带注释说明何时用]

## [环境特殊规范章节名 — 按实际情况命名]

[仅当项目有特殊部署环境约束时才写。如 NFS 操作、多机一致性、容器内执行等]

## 核心架构：[核心业务链路名]

[项目最关键的业务执行链路，需要跨多个文件才能理解的"大图"。
用流程图或文字链表示。附关键文件映射表。]

## 编码规范

[从代码中观察到的实际约定，不写通用最佳实践。
包含：类型注解/docstring/导入顺序/日志格式/异常处理 的具体模式]

## 三层边界模型

### ✅ 必须执行
[在本项目中必须遵守的硬性规则]

### ⚠️ 需先询问
[改之前必须确认的操作]

### ❌ 禁止操作
[绝对不能做的事情，基于项目踩坑经验]

## 测试要求

[测试框架/运行方式/覆盖率要求/验证标准]

## [可选：病例库/知识库系统]

[仅当项目有问题追踪机制时才写]

---
**版本**: v1.0
**最后更新**: [当前日期]
**维护者**: [项目团队]
```

## 裁剪规则

根据项目规模和特点决定写哪些章节：

| 项目特征 | 需要的章节 | 跳过的章节 |
|---------|-----------|-----------|
| 小型库（<20 文件） | 身份+命令+编码规范 | 核心架构、病例库 |
| 中型应用 | 全部核心章节 | 病例库（除非已有） |
| 大型多机部署 | 全部章节 + 环境规范 | — |
| Monorepo | 根级索引 + 各模块子 CLAUDE.md | 合在一起会太长 |

## 质量检查清单

生成完成后自检：

- [ ] 没有写"使用有意义的变量名"这种废话
- [ ] 没有重复 README.md 的内容
- [ ] 每条规范都有具体文件路径或命令支撑
- [ ] 技术栈版本号从配置文件中实际读取，不是猜的
- [ ] 命令都是实际可运行的（从 Makefile/scripts 中提取）
- [ ] 没有列出可以 `ls` 一下就知道的文件结构
- [ ] 三层边界模型的"禁止"项都来自项目实际踩坑
- [ ] 环境特殊规范章节有代码示例（正确做法 vs 禁止做法）

## 增量更新模式

如果已存在 CLAUDE.md：

1. 读取现有内容，标记每个章节的信息来源
2. 扫描代码库变化（新增模块/变更架构/新增约束）
3. 只更新过期或缺失的部分
4. 在文档末尾更新版本号和日期
5. 输出变更摘要：新增了什么、更新了什么、删除了什么

## 与内置 /init 的差异

| 维度 | 内置 /init | 本 skill |
|------|-----------|---------|
| 深度 | 浅层文件结构 | 深度架构+环境+约束 |
| 环境感知 | 无 | NFS/Docker/多机/时区 |
| 边界模型 | 无 | 三层（必须/确认/禁止）|
| 测试策略 | 通用提及 | 具体框架+运行命令+覆盖率 |
| 病例库 | 无 | 支持（可选） |
| 增量更新 | 覆盖重写 | 差异更新+版本追踪 |
| 输出风格 | 通用最佳实践 | 仅项目特定事实 |
