# 病例库系统设置指南

病例库是可选章节，仅当项目需要持久化问题诊断经验时才在 CLAUDE.md 中添加。

## 适用场景

- 复杂的多机部署环境（问题容易反复出现）
- 团队多人协作，需要共享排查经验
- 有外部依赖的项目（第三方 API/CLI 工具版本升级踩坑）
- 长期维护的项目（历史问题需要追溯）

## 目录结构

```
.problems/
├── INDEX.md          # 病例索引（按模块/症状分类）
├── nfs-stale-read.md
├── timezone-mismatch.md
└── docker-network.md
```

## 病例模板

每份病例 6 个二级标题：

```markdown
# [简短标题]

## 症状
[用户/系统表现出的异常行为]

## 病因
[根本原因分析]

## 检验报告
[查了什么 → 看到什么 → 得出什么结论]
- `grep -rn "xxx" app/` → 找到 3 处使用旧 API
- `docker logs cursor-server | grep ERROR` → 发现连接超时
- `git log --oneline -5 app/services/` → 最近改动了 xxx

## 关键检查点
[下次遇到类似问题时应该先检查什么]
1. 检查 xxx 配置
2. 确认 yyy 版本
3. 验证 zzz 连通性

## 治疗方案
[具体修复步骤 + 代码变更]

## 就诊历史
[只追加不覆盖，带日期和 commit hash]
- 2026-04-03 (abc1234): 首次发现，修复 xxx
- 2026-04-10 (def5678): 复发，根因是 yyy 未同步更新
```

- ...

- ...

## CLAUDE.md 中的病例库章节

```markdown
## 病例库（`.problems/`）

持久化问题诊断经验，三层架构：

| 层 | 内容 | 维护者 |
|----|------|--------|
| Schema | CLAUDE.md 本节定义 | 团队 |
| Wiki | `.problems/INDEX.md` + 各病例 | AI 写、人审 |
| Raw | 日志、diff、对话上下文 | 人提供 |

**使用流程**:
- **问诊**: 遇 bug 先读 `.problems/INDEX.md` 按症状匹配
- **归档**: 解决后判断追加还是新建，写入后执行巡检
- **巡检**: 大重构后全量检查路径有效性

**与其他机制的分工**:
| 病例库 | 具体诊断、检验、治疗方案 |
| Git history | 代码变更事实 |
| Memory | 用户偏好、协作风格 |
| 业务文档 | 当前系统行为 |
```

## 何时建议添加病例库

在初始化 CLAUDE.md 时，如果发现以下信号，建议用户考虑添加病例库：

1. 已有 `.problems/` 或类似目录 → 直接写入 CLAUDE.md
2. git log 中有大量 "fix:" 提交且部分是反复修同一类问题 → 建议添加
3. 项目有复杂外部依赖（CLI 工具/第三方服务） → 建议添加
4. 多机/分布式环境 → 强烈建议添加
5. 简单的库/工具项目 → 不需要，不要过度工程化
