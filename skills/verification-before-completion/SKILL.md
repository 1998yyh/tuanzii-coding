# 完成前验证

## 概述
> 【老王注】全篇一句话：先说证据再下结论。没跑命令就敢说"通过了"，这不是效率是撒谎。
> 【老王注】"违反字面就是违反精神"——别琢磨怎么钻措辞的空子，没用。

没有验证就声称工作已完成，是不诚实，不是高效率。

**核心原则：**永远先有证据，再作声明。

**违反这条规则的字面要求，同样是违反其精神。**

## 铁律

```
没有最新验证证据，不得声明完成
```

若未在本条消息中运行验证命令，就不能声称它已通过。
> 【老王注】注意"in this message"：上一轮跑过不算数，结论必须用刚出炉的输出撑着。

## 验证关卡
> 【老王注】这五步是执行核心：找命令 → 完整跑 → 读完输出看退出码 → 对照结论 → 才能开口。
> 【老王注】跳任何一步都不算验证，就是撒谎——包括"跑了一半看着差不多"。

```
在声称任何状态或表达满意之前：

1. **识别：**什么命令能够证明这个声明？
2. **运行：**执行完整的命令，且必须是刚刚执行、完整运行的结果。
3. **阅读：**阅读完整输出，检查退出码并统计失败数。
4. **核实：**输出是否支持该声明？
   - 否：带着证据说明实际状态。
   - 是：带着证据作出声明。
5. **最后才：**作出声明。

跳过任一步都不是验证，而是在撒谎。
```

## 常见误区

| 声明 | 需要 | 不足以证明 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 之前的运行结果、“应该会通过” |
| Linter 无问题 | Linter 输出：0 个错误 | 局部检查、推断 |
| 构建成功 | 构建命令退出码为 0 | Linter 通过、日志看起来正常 |
| 缺陷已修复 | 原始症状的测试通过 | 代码已改、假定已修复 |
| 回归测试有效 | 已验证红绿循环 | 测试只通过过一次 |
| Agent 已完成 | 版本控制差异显示改动 | Agent 报告“成功” |
| 需求已满足 | 逐项检查清单 | 测试通过 |
> 【老王注】右边一列全是自欺套路：拿"上次的结果""部分的检查""别人的汇报"当证据，一条都不作数。
> 【老王注】"测试过了"不等于"需求做完了"——测试只证明你测过的那部分。

## 红旗：立即停止
> 【老王注】"should / probably / seems to"是红旗词——你一旦这么措辞，就说明你没验证，立即停手去跑命令。
> 【老王注】"就这一次""太累了"也是红旗：想偷懒的念头本身就是违规前兆。

- 使用“应该”“大概”“看起来”等措辞
- 验证前表达满意，例如“很好”“完美”“完成了”
- 未验证就准备提交、推送或创建 PR
- 相信 Agent 的成功报告
- 依赖局部验证
- 想着“就这一次”
- 因疲劳而想尽快结束工作
- **未运行验证却使用任何暗示成功的措辞**

## 防止自我辩解
> 【老王注】左边是借口，右边是拆穿。自信、疲劳、"换个说法"，没有一样能替代跑出来的输出。

| 借口 | 事实 |
|--------|---------|
| “现在应该能用了” | 运行验证 |
| “我很有把握” | 把握不等于证据 |
| “就这一次” | 没有例外 |
| “Linter 通过了” | Linter 不等于编译器 |
| “Agent 说成功了” | 独立验证 |
| “我累了” | 疲惫不是借口 |
| “局部检查就够了” | 局部结果不能证明整体 |
| “我换个说法，规则就不适用” | 精神优先于字面 |

## 关键模式
> 【老王注】五类高频场景的合格说法与违规说法。汇报句式照抄 ✅ 行：先贴命令和输出，再下结论。
> 【老王注】回归测试必须见到红再见到绿： revert 修复后测试必须失败，否则这个测试什么都没验证。
> 【老王注】子 agent 说"成功了"不算数，自己去看 diff 确认。

**测试：**
```
✅ [Run test command] [See: 34/34 pass] "All tests pass"
❌ "Should pass now" / "Looks correct"
```

**回归测试（TDD 红绿循环）：**
```
✅ Write → Run (pass) → Revert fix → Run (MUST FAIL) → Restore → Run (pass)
❌ "I've written a regression test" (without red-green verification)
```

**构建：**
```
✅ [Run build] [See: exit 0] "Build passes"
❌ "Linter passed" (linter doesn't check compilation)
```

**需求：**
```
✅ Re-read plan → Create checklist → Verify each → Report gaps or completion
❌ "Tests pass, phase complete"
```

**Agent 委派：**
```
✅ Agent reports success → Check VCS diff → Verify changes → Report actual state
❌ Trust agent report
```

## 为什么重要
> 【老王注】代价不是理论：信任一旦破了很难补，"我不信你"这句话比任何 bug 都贵。

来自 24 次失败记录：
- 人类协作者说“我不相信你”，信任已经破裂
- 未定义函数被交付，可能导致崩溃
- 缺失的需求被交付，功能不完整
- 虚假的完成状态会造成时间浪费，继而返工
- 这违反了“诚实是核心价值；撒谎就会被替代”的原则

## 何时应用
> 【老王注】适用范围是"任何暗示成功的表述"，不只是"完成了"三个字——换个说法照样触发本规则。

**在以下情况之前始终应用：**
- 任何形式的成功或完成声明
- 任何满意表达
- 任何关于工作状态的正向表述
- 提交、创建 PR、完成任务
- 转入下一项任务
- 向 Agent 委派任务

**规则适用于：**
- 原话
- 改写和同义表达
- 成功的暗示
- 任何暗示完成或正确的沟通

## 最终准则
> 【老王注】收尾九个字：跑命令、看输出、再下结论。顺序不能反。

**验证没有捷径。**

运行命令，阅读输出，然后才能声明结果。

这点不可协商。
